DeskBridge USB / BLE Command Protocol
=====================================

Documento del protocolo actual entre:
- PC App <-> CHIP_CORE por USB CDC Serial.
- Mobile App <-> CHIP_CORE por BLE GATT.
- CHIP_CORE <-> CHIP_KEYPAD por UART interno.

Firmware documentado:
- Firmware: 0.4.0
- Serial protocol: 0x04
- USB VID/PID: 0x1209 / 0xDB01
- USB bcdDevice: 0x0400


1. Canal USB fisico
===================

Canal:
- USB CDC principal.
- Un solo puerto COM visible en Windows.
- Comandos ASCII en texto plano.
- Cada comando termina con salto de linea.
- El prompt `db>` pertenece a la shell del firmware.

Formato enviado por el PC:
  [HEADER] comando argumentos

Ejemplos:
  [SHL] ping
  [CMD] program?
  [DAT] display fps 30


2. Headers PC -> CHIP_CORE
==========================

[SHL]
  Comando de shell interactivo. Lo usa la terminal CLI humana.

[CMD]
  Comando directo invocado por programa, GUI, TUI o llamada automatizada.

[DAT]
  Escritura/consulta de datos de configuracion.
  Actualmente el firmware interpreta el payload con el mismo parser de comandos.

[STK]
  Paquete tipo diccionario.
  Reservado para datos estructurados futuros.

[STE]
  Escritura/consulta de estado.
  Actualmente el firmware interpreta el payload con el mismo parser de comandos.

[INF]
  Consulta informativa.
  Actualmente el firmware interpreta el payload con el mismo parser de comandos.

[CNF]
  Configuracion heredada.
  Reservado.

[PRG]
  Programa/interfaz futura.
  Reservado.


3. Headers CHIP_CORE -> PC
==========================

[RSP]
  Respuesta normal.

[ERR]
  Error con codigo de 32 bits.

[MRK]
  Marcador interno no critico.

[LOG]
  Mensaje que el programa de PC debe guardar en log.

[EVT]
  Evento del firmware.

[TSK]
  Peticion de tarea al PC.

[DAT]
  Dato o variable.

[STE]
  Estado tecnico.

[STK]
  Paquete de variables tipo diccionario.

[INF]
  Informacion generica.


4. Formato de respuesta
=======================

Respuesta normal:
  [RSP] grupo: clave:valor, clave:valor

Error:
  [ERR] 0x00000000 descripcion

Marcador:
  [MRK] 0x00000000 descripcion

Log:
  [LOG] descripcion

Notas de parseo:
- El grupo de una respuesta `[RSP]` es el texto antes del primer `:`.
- Los pares de datos usan formato `clave:valor`.
- Un comando puede devolver varias lineas `[RSP]`.
- El PC debe leer hasta el prompt `db>`.

12. BLE movil
=============

El ESP32-S3 trabaja como periferico BLE.

Advertising:
- Nombre por defecto: DBx01
- Si Wireless BLE esta habilitado, el firmware anuncia el servicio.
- Al desconectar un cliente BLE, el firmware reinicia advertising.
- La conexion automatica por cercania la debe implementar la app movil como
  cliente BLE central, recordando el dispositivo y conectando al detectarlo.

Servicio BLE GATT:
  Device name: DBx01
  Service UUID: 8f7f0001-6b21-4f5e-9c7e-123456780001
  Characteristic UUID: 8f7f0002-6b21-4f5e-9c7e-123456780001
  Properties: READ, WRITE, NOTIFY

Notificaciones BLE periodicas:
  [INF] device: name:DeskBridge chip_core, firmware:0.4.0, protocol:0x04, product_id:00112233445566778899AABB, uptime_s:12
  [DAT] sensors: temp_c:24.1, humidity_pct:45.0, co2_ppm:620, lux:80.5, tvoc_ppb:12, aqi:1
  [DAT] config: sample_ms:6000, samples:1, temp_unit:0, display_design:0, display_fps:20, usb:mounted
  [DAT] light: enabled:yes, mode:Composite, brightness:16384, kelvin:32767, cold_min:0, cold_max:65535, warm_min:0, warm_max:65535

Comandos BLE por escritura en la characteristic:
  info
  info?
  program
  program?
  sensors
  sensors?
  sensor
  sensor?
  config
  config?
  light
  light?
  light enabled [on|off|1|0|true|false|yes|no]
  light brightness [0..65535]
  light kelvin [0..65535]
  light mode [0|1|2|one|two|composite|simple|dual]
  light sync

Respuestas BLE:
  [INF] device: ...
  [DAT] sensors: ...
  [DAT] config: ...
  [DAT] light: ...
  [ERR] ble.command.unknown
  [ERR] ble.light.disabled
  [ERR] ble.light.enabled.invalid
  [ERR] ble.light.brightness.invalid

Restricciones BLE:
- `light brightness`, `light kelvin` y `light mode` solo se aceptan si la luz
  esta habilitada.
- Para habilitarla desde el movil:
  light enabled on


13. UART interno CORE -> CHIP_KEYPAD
====================================

Resumen fisico:
- UART2 TX: PA2
- UART2 RX: PA3
- EVENT: PA9
- STATE: PA8
- RST: PA10
- Baudrate UART: 115200
- Timeout UART interno CORE -> KEYPAD: 35 ms

El PC no habla directamente con CHIP_KEYPAD.
El PC envia comandos USB al CORE y el CORE traduce parte de ellos al protocolo
UART interno del KEYPAD.

Comandos UART internos:
  PING
  INFO?
  LIGHT?
  POWER?
  EVENT?
  BUTTON ASSIGN?
  EVENT CLEAR
  CONFIG DEFAULTS
  LIGHT SET ENABLED [0|1]
  LIGHT SET MODE [0|1|2]
  LIGHT SET BRIGHTNESS [0..65535]
  LIGHT SET KELVIN [0..65535]
  LIGHT SET COLD_MIN [0..65535]
  LIGHT SET COLD_MAX [0..65535]
  LIGHT SET WARM_MIN [0..65535]
  LIGHT SET WARM_MAX [0..65535]
  LIGHT SET BRIGHTNESS_STEP [value]
  LIGHT SET KELVIN_STEP [value]
  BUTTONS SET CLICK_MS [ms]
  BUTTONS SET LONG_MS [ms]
  BUTTONS SET REPEAT_MS [ms]
  BUTTON SET [0..4] ACTION [id]

Respuesta UART interna esperada:
  OK KEY=value KEY=value


14. Reglas para el software del PC
==================================

1. Enviar comandos interactivos con `[SHL]`.
2. Enviar comandos directos/programaticos con `[CMD]`.
3. Leer hasta `db>`.
4. Parsear primero el header de cada linea.
5. En `[RSP]`, usar el texto antes de `:` como grupo.
6. Parsear pares `clave:valor`.
7. Tratar `[ERR] 0x00000000 descripcion` como error estructurado.
8. Guardar `[LOG]` en el log del PC.
9. Mostrar `[MRK]` solo en modo debug o diagnostico.
10. Mantener compatibilidad con varias lineas por comando.
